## ✨ 主要機能 (Key Features)

- **📧 AIメール・予定の要約**

  GmailとGoogle Calendarの予定をGPT-4o-miniが解析し、重要事項・アクションアイテム・期限・緊急度を自動抽出します。

- **🧹 2層AIフィルタリング**

  gpt-4o-mini（安全性重視モデル）を2段階構造にしてメールを選別。重要メール見落とし0件を維持しつつ、スパム混入を低減。ユーザーによる「AIチューニング」でさらに精度向上。

- **🎯 AIチューニング**

  メールへのKEEP/SKIPアノテーションとプロフィール設定で、AIの判断基準を自分好みに育てられます。ラベル付け内容は自動で"事前フィルタリング"と"Few-Shotプロンプト"に反映されます。

- **🔊 音声ブリーフィング**

  要約内容をOpenAI API TTS / EdgeTTSで自然な日本語音声に変換。通勤中や家事の合間に「聴く」だけで一日の情報整理が完了します。

- **⏰ スケジュール自動実行**

  毎朝指定した時間に自動でブリーフィングを生成し、Google Drive / S3に保存。設定後は完全にハンズフリーで運用可能。

利用環境に合わせて、以下の3つの方法から選択してください。

### 1. 🐳 Docker で開始する (推奨)

最も簡単かつ確実に実行できる方法です。PCにDockerがインストールされている必要があります。
（※初回クローン時は `node_modules` や `dist` もビルド時に自動生成されます）

#### 事前準備

起動前に以下の2つの外部サービスを設定してください。

**① Google Cloud Console（Gmail / Google Calendar / Google Drive 連携用）**

1. [Google Cloud Console](https://console.cloud.google.com/) でプロジェクトを作成
2. 「APIとサービス」→「ライブラリ」から以下を有効化
   - Gmail API
   - Google Calendar API
   - Google Drive API
3. 「APIとサービス」→「OAuth同意画面」→「概要」で必要な情報を入力
   - ユーザーの種類：外部
   - アプリ名：(任意)
   - サポートメールアドレス：[EMAIL_ADDRESS]
   - 開発者連絡先：[EMAIL_ADDRESS]
4. 「APIとサービス」→「認証情報」→「OAuth 2.0 クライアント ID」を作成
   - 種類: ウェブアプリケーション
5. 「承認済みのリダイレクト URI」に以下を追加
   - `http://localhost:8000/api/oauth/google/callback`
6. 表示された **クライアントID** と **クライアントシークレット** を控える
7. 「APIとサービス」→「OAuth同意画面」→「対象」でテストユーザーを設定
   - 公開ステータスを「テスト」にする
   - テストユーザーのメールアドレスを入力

**② AWS Cognito（ログイン認証用）**

1. [AWSコンソール](https://console.aws.amazon.com/) で Cognito ユーザープールを作成
2. アプリクライアントを作成し、コールバックURLに `http://localhost:3000` を設定
3. 以下の値を控える
   - ユーザープール ID（例: `ap-northeast-1_xxxxxxxxx`）
   - アプリクライアント ID
   - Cognito ドメイン（例: `https://your-domain.auth.ap-northeast-1.amazoncognito.com`）

## 🤖 プロンプトエンジニアリング (Prompt Engineering)
本アプリのデイリーブリーフ生成では、Gmailから取得したメールをメールアドレスで事前フィルタリングした後、OpenAI GPTでメール本文を検査・フィルタリング・要約する**4層フィルタリング**を採用している。

メールのフィルタリングでは、スパムの除外よりも重要メールの除外を避けることを重視している。

### ●使用する混合行列

| 実際の判定\AIの判定 | KEEP | SKIP |
| --- | --- | --- |
| KEEP | TP | FN |
| SKIP | FP | TN |

> 再現率 = TP / (TP + FN)　# 実際にKEEPすべきメールをどれだけKEEPできたか※重要  
> 適合率 = TP / (TP + FP)　# AIがKEEPしたメールのうち、実際にKEEPすべきものはどれだけあったか  
> F1 = 2 * (適合率 * 再現率) / (適合率 + 再現率)　# KEEPとSKIPのバランス


### ●背景

初期実装では、スパムメールが通過してしまう問題（FP 多数）と、
重要メールを誤って除外してしまう問題（FN 発生）が混在していた。

### ●解決プロセス

**① 評価指標の設計**
FNはユーザーが気づけない致命的な欠損のため、再現率（FN最小化）を最優先指標に設定。F1スコアで総合評価。

**② Few-Shot による人力改善**
Few-Shotに可能性を感じ、人力でプロンプトを書き直す改善を繰り返した。しかし手作業での改善には限界があり、スクリプトによる反復テストへ移行することを決定。

**③ 合成テストデータの作成**
実メールに依存しない再現可能な評価環境を構築するため、境界ケースを含む400件の合成メールデータを生成。

**④ ベースライン計測 → 反復改善（19ラウンド・57回以上のテスト）**
v1プロンプトでF1・FP・FNを計測し出発点を定量化。プロンプト変更 → テスト実行 → 計測 → 原因分析のサイクルを繰り返した。
改善の中で「1つのモデルで安全性と精度を同時に最適化することは困難」という構造的な問題を発見。

**⑤ 2層構造の採用**
Layer1（安全性重視：FNを出さない）とLayer2（精度重視：FPを削る）に役割を分離する設計に転換。
Layer1のFNを増やさずにFPのみを大幅削減することに成功。

**⑥ 汎化検証**
新規パターン55件で過学習チェックを実施し、未知データへの汎化を確認。

**⑦ カスタマイズ性の向上**
ユーザーごとに精度を高めるため、アノテーション（KEEP/SKIP手動ラベル）と条件追加（important_senders / keep_feature など）をプロンプトへ反映できる仕組みを実装。
また、プロンプトへの直書きでは構造が崩れやすいため、`.replace()` による注入方式に変更し、ステップ構造を安定して維持できるようにした。


### アプリが抱える問題点
**機密保持のため削除**



### ●解決アプローチ：4層フィルタリングアーキテクチャ

```
全受信メール
    ↓
【事前フィルタリング】  役割: 事前にカスタマイズされた「SKIPする送信元メールアドレス」からのメールを除去

    ↓ スパムメールの80~90%を除去

【Layer 1: gpt-4o-mini】 役割: 安全性重視のモデル（重要メールを絶対に SKIP しない）

    ↓ スパムメールの除去と境界データの安全な判定(50%~90%の不要なメールを除去)

【Layer 2: gpt-4o-mini】     役割: 精度重視のモデル（残りのスパムを積極的に除去する）

    ↓ 不要なメールの厳格な判定と除去

【全文整形: gpt-5-mini】 役割: 重複した内容をまとめる、AIから出力された不要な情報の排除、人間が理解しやすいように全文を整形する

    ↓ 音声化

最終出力
```

### ●主な教訓

- **条件の追加・変更は慎重に行う** — 新条件を加えたせいで既存の正しい分類が崩れることが繰り返し発生  
  **解決策:** 層ごと、ステップごとに役割を分担させることでAIの混乱を防ぎ、トレードオフの影響を最小限に抑えた

- **ポジティブリストは危険** — KEEP 条件を先に定義したところ、スパムも条件を満たし F1 が 0.030 に激減  
  **※4層フィルタリング**でKEEPステップとSKIPステップを厳格に区分することで改善。現在はポジティブリストも使用

- **Few-shot の圧倒的効果**
   — SKIP・KEEP それぞれの具体例を提示することで、境界線上のケース（議事録だが依頼事項がある等）の判定精度が劇的に向上。

- **AIは相対的な概念（時系列）を理解するのが苦手**
   — 「本日以降」という相対的な概念を理解するのが苦手なため、絶対的な日付に変換して判定させることで改善。

- **AIのモデルによって性能に明確に差がある**
   — gpt-4o-miniで「原則KEEP」ルールを導入したところ、ダミーデータ、実データともにFNが0件で安定。しかし、代わりにFPが多くなる傾向があり、フィルタの目は荒いと言える。
   — 逆に、gpt-o4-mini, gpt-5-miniではFPがほとんど無く、出力はgpt-4o-miniよりスリムに見えるが、ダミーデータでの検証ではFNが0~5%程度発生し続けた。

## 🔍 詳細な設計について

### 1. 2層フィルタリングアーキテクチャの設計と評価基盤の構築

**課題**: メールフィルタリングでは「重要メールを見逃さない（再現率）」と「スパムを除外する（適合率）」が相反する。単一モデル・単一プロンプトではどちらかが必ず犠牲になり、バランスが取れなかった。

**解決**: Layer1（gpt-4o-mini・recall重視）× Layer2（gpt-4o-mini・precision重視）の2段階構成を設計。「重要メールを絶対に SKIP しない」Layer1 で安全網を張り、通過したメールを Layer2 で再判定してスパムを積極除去する役割分担を明確化。各層でプロンプト設計を独立させることで、一方の改善がもう一方に干渉しない構造とした。

定量評価のために手動ラベル付き116件＋LLM生成55件の合成データセット（計171件）を構築し、混同行列（TP/TN/FP/FN）・F1スコアを指標に60回以上のテストサイクルを実施。「改善が再現できる評価基盤」ごと設計することで、プロンプト変更の影響を定量的に追跡できるようにした。

---

### 2. ポーリング処理とバックグラウンドタスク（ジョブ管理）

**課題**: 日報生成はメール取得・AI要約・TTS変換・Drive アップロードを含む重い非同期処理であり、HTTPレスポンスのタイムアウト内に完了しない。フロントエンドがリアルタイムで進捗を把握する手段が必要だった。

**解決**: `daily_brief_jobs` テーブルでジョブのライフサイクル（`PENDING → PROCESSING → COMPLETED / FAILED`）を管理する非同期ジョブ方式を採用。生成リクエストを受けると FastAPI の `BackgroundTasks` でバックグラウンド処理を起動し、即座にジョブIDを返す。フロントエンドはそのジョブIDを使って `GET /daily-brief/status/{job_id}` を数秒おきにポーリングし、`COMPLETED` になった時点で結果を受け取る設計とした。`COMPLETED` / `FAILED` のジョブはポーリング応答後にDBから即削除し、DB肥大化を防いでいる。また APScheduler を用いてユーザーが設定したスケジュール時刻（JST）に自動生成を起動する cron ジョブも実装し、手動・自動の両トリガーに対応した。

**並列処理の制御（asyncio.Semaphore）**: 外部API呼び出しはすべて `asyncio.gather` による並列実行だが、API レート制限に引っかからないよう処理ごとにセマフォで同時実行数を制限している。

| 処理 | セマフォ上限 | 理由 |
|------|-----------|------|
| TTS チャンク生成 | 5 | OpenAI TTS の API レート制限対策 |
| Gmail スレッド取得 | 20 | Gmail API のレート制限に収まる範囲で高速化 |
| Gmail メッセージ取得 | 20 | 同上 |

### 3. Google Limited Use・GDPR の遵守とプライバシーポリシーの策定

**課題**: Google OAuth で Gmail・Calendar スコープを利用する場合、Google API Services User Data Policy（Limited Use 要件）の遵守が義務付けられる。取得したユーザーデータの利用目的を明示し、第三者提供禁止・保存最小化などの要件を満たす必要があった。また GDPR の「忘れられる権利」への対応も求められた。

**解決**: プライバシーポリシーを策定・公開し、Gmail・Calendar データの利用目的をブリーフィング生成のみに限定する旨を明記。`DELETE /users/me` エンドポイントで OAuthトークン・プロンプトテンプレート・ユーザーレコード・S3 音声ファイル・Cognito アカウントをまとめて削除できるデータ削除フローを実装し、GDPR の「忘れられる権利」に対応した。OAuth アプリの審査申請では、実際のアプリ動作のスクリーンキャプチャや Limited Use 準拠の説明文を Google に提出した。

---

### 4. Google OAuth 認証の試行錯誤（公開設定・マルチアカウント方式）

認証方式について、当初は「1ユーザー = 1 Google アカウント」のシンプルな構成から着手し、その後「Gmail・Calendar・Drive をそれぞれ別アカウントで接続できるマルチアカウント方式」へ段階的に設計を変更した。その後「メインアカウント＋サブアカウント（閲覧専用など）」という階層型の権限分離方式に落ち着いた。これらの試行錯誤を経て、`oauth_tokens` テーブルを `(user_id, provider, provider_email)` の複合キーで管理する現在の設計にたどり着いた。

---

### 5. タイムゾーンに伴うデータ不整合の解消（UTCへの完全統一）

**課題**: システム内で扱う日時データにおいて「タイムゾーン情報を持つデータ（Aware datetime）」と「持たないデータ（Naive datetime）」が混在しており、スケジュールや通知時刻などの比較・演算処理を行う際に `TypeError: can't compare offset-naive and offset-aware datetimes` が頻発した。

**解決**: アプリケーション層およびデータベース層での日時の取り扱いを「すべてUTC（協定世界時）」に統一するルールをスプリントの途中で徹底した。システムの内部処理でタイムゾーンを変換するロジックを全面的に排除し、データベースへの保存からバックエンドでの内部演算までをすべてタイムゾーン情報を持たないUTC日時で行うアーキテクチャに変更した。これにより、バグの温床であった日付比較時の不整合を根本から解消した。

### Docker マルチステージビルド

ビルドを2段階に分けることで、**フロントエンドとバックエンドを1つのDockerイメージに同梱**しています。
最終イメージには Stage 2 だけが残るため、Node.js の開発依存が混入せずイメージサイズを最小化できます。

```
Stage 1 (node:18-alpine)
  npm install && npm run build
  → React アプリを dist/ にビルド
        |
        | dist/ をコピー
        ↓
Stage 2 (python:3.11-slim)          ← 最終イメージはこれだけ
  pip install（Python依存）
  dist/ を /app/static/ に配置
  .env を削除（機密情報の混入防止）
  uvicorn で API + 静的ファイルを同時配信
```

フロントとバックが1イメージに同梱されるため、**デプロイは `deploy_backend.ps1` の1コマンドのみ**で完結します。
